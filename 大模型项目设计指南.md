## 一、设计思路与核心原则

### 1.1 项目定位原则

**明确边界，不贪大求全**

```
❌ 错误：做一个万能的教育AI助手
✅ 正确：做一个课程咨询客服，解决"有什么课、学什么内容"问题
```

**核心原则**：
- **单一职责**：一个系统解决一个明确的问题
- **数据驱动**：先确定有什么数据，再设计能回答什么问题
- **渐进迭代**：先跑通基础流程，再逐步增强

### 1.2 用户交互设计原则

**隐藏复杂性，暴露友好性**

| 层面 | 用户看到 | 系统内部 |
|------|---------|---------|
| 交互 | 自然对话 | Agent决策、工具调用 |
| 回答 | 直接答案 | 多路检索、结果融合 |
| 过程 | 无感知 | 实体对齐、向量化、图谱查询 |

**黄金法则**：
> 用户问"有Python课程吗？"，系统内部可能经过10个步骤，但用户只应该看到一句话回答。

### 1.3 数据分层原则

```
原始数据层（MySQL 业务数据库）
    ↓ ETL 清洗、结构化
知识存储层（Neo4j 图谱 + ChromaDB 向量）
    ↓ 索引构建
检索服务层（Cypher 查询 + 向量相似度）
    ↓ 查询、匹配
应用服务层（Agent 决策 + LLM 生成）
    ↓ SSE 流式输出
用户交互层
```

**MySQL 的核心地位**：MySQL 是整个系统的**唯一原始数据源**。Neo4j 知识图谱和 ChromaDB 向量库的数据均通过 ETL 流程从 MySQL 导入：
- `data_prepare.py`：从 MySQL 提取课程、章节、用户行为等数据，清洗后导入 Neo4j
- `document_ingestion.py` / `create_smart_edu.py`：从 MySQL 提取文本字段，向量化后导入 ChromaDB
- `entity_mapping` 表：存储同义词到标准词的映射，支撑实体对齐功能

## 二、架构设计流程

### 2.1 标准设计流程（5步法）

```
Step 1: 需求分析
    ├── 用户会问什么问题？
    ├── 需要哪些数据来回答？
    └── 回答的准确度要求多高？
    
Step 2: 数据盘点
    ├── 已有数据：课程表、章节表...
    ├── 需要补充：课程介绍文档、FAQ...
    └── 数据质量：是否清洗、标准化？
    
Step 3: 技术架构设计
    ├── 数据存储：关系型 + 图 + 向量
    ├── 检索策略：关键词 + 语义 + 图谱
    └── 生成策略：直接回答 / RAG增强
    
Step 4: 提示词设计
    ├── 系统提示词（角色定义）
    ├── 工具提示词（功能说明）
    └── 约束条件（禁止事项）
    
Step 5: 迭代优化
    ├── 测试用例设计
    ├── Bad Case分析
    └── 提示词调优
```

### 2.2 架构模式选择

#### 模式A：纯LLM（最简单）
```
用户问题 → LLM → 回答
```
**适用**：通用问答，不需要特定数据
**缺点**：幻觉严重，无法保证准确性

#### 模式B：RAG检索增强（推荐）
```
用户问题 → 向量检索 → LLM生成回答
```
**适用**：有文档知识库，需要基于文档回答
**优点**：减少幻觉，可追溯来源

#### 模式C：Agent + 多工具（本项目使用）
```
用户问题 → Agent决策 → 多工具并行 → 结果融合 → LLM生成
```
**适用**：复杂场景，需要多种数据源
**优点**：灵活、可扩展、准确度高

#### 模式D：知识图谱 + LLM（高级）
```
用户问题 → 实体抽取 → 图谱查询 → 推理 → LLM生成
```
**适用**：关系复杂、需要推理的场景
**优点**：可解释性强、支持复杂推理

### 2.4 记忆与上下文设计

**三层记忆架构**：

```
HTTP Session (Cookie) ──→ thread_id ──→ InMemorySaver
     │                         │               │
  浏览器端                   Agent配置        内存中的对话栈
  session_id=uuid            configurable     {messages, steps}
                             {"thread_id": id}
```

| 层级 | 实现 | 作用 | 生命周期 |
|------|------|------|----------|
| **Session 层** | FastAPI `SessionMiddleware` + Cookie | 识别同一用户/浏览器 | Cookie max_age=3600s |
| **Thread 层** | LangChain `configurable={"thread_id": session_id}` | 索引对话历史 | 与 Session 绑定 |
| **Storage 层** | `InMemorySaver` | 存储消息列表和中间状态 | 进程内存，服务重启丢失 |

**关键机制**：
- Agent 每次只接收当前用户消息，自动从 `InMemorySaver` 加载同 `thread_id` 的历史状态
- `ToolMessage` 对前端隐藏，但对 LLM 完全可见，确保生成基于真实检索结果
- **局限性**：当前为内存级存储，生产环境应替换为 `PostgresSaver` 或 `RedisSaver`

## 三、技术选型决策

### 3.1 数据存储选型

| 数据类型 | 选型 | 理由 |
|---------|------|------|
| 业务数据 | MySQL | 成熟稳定，支持复杂查询 |
| 关系网络 | Neo4j | 原生图数据库，Cypher查询直观 |
| 文档向量 | ChromaDB | 轻量级，Python原生支持 |
| 会话缓存 | 内存/Redis | 简单场景用内存，生产用Redis |

### 3.2 模型选型

| 用途 | 选型 | 部署方式 |
|------|------|---------|
| 嵌入模型 | BGE-base-zh-v1.5 | 本地部署（CPU/GPU） |
| 大语言模型 | DeepSeek | API调用 |
| 备选模型 | GPT-3.5/文心一言 | API调用 |

**选型原则**：
- 嵌入模型本地部署（调用频繁，需要低延迟）
- LLM使用API（复杂度高，自部署成本高）

### 3.3 框架选型

| 层级 | 选型 | 理由 |
|------|------|------|
| Web框架 | FastAPI | 异步支持好，自动生成文档 |
| LLM框架 | LangChain | 生态完善，Agent支持好 |
| 向量库 | ChromaDB | 与LangChain集成好 |

## 五、RAG架构实现

### 5.1 RAG核心流程

```
用户问题
    ↓
┌─────────────────┐
│ 1. 查询理解     │ ← 意图识别、实体抽取
└────────┬────────┘
         ↓
┌─────────────────┐
│ 2. 检索阶段     │ ← 多路并行检索
│ ├─ 向量检索     │
│ ├─ 关键词检索   │
│ └─ 图谱查询     │
└────────┬────────┘
         ↓
┌─────────────────┐
│ 3. 重排序/过滤  │ ← 相关性评分、去重
└────────┬────────┘
         ↓
┌─────────────────┐
│ 4. 上下文构建   │ ← 组装Prompt
└────────┬────────┘
         ↓
┌─────────────────┐
│ 5. LLM生成      │ ← 生成最终回答
└─────────────────┘
```

### 5.2 检索策略选择

| 策略 | 适用场景 | 实现方式 |
|------|---------|---------|
| 纯向量检索 | 语义相似度要求高 | Embedding + 向量库 |
| 纯关键词检索 | 精确匹配要求高 | 全文索引 |
| 混合检索 | 兼顾语义和精确 | 向量 + 关键词，加权融合 |
| 图谱检索 | 关系推理 | Cypher查询 |

### 5.3 本项目RAG实现

**数据流转路径**：

```
MySQL (course_info, chapter_info...)
    ├──→ data_prepare.py ──→ Neo4j (Course, Chapter, Teacher 节点)
    │
    ├──→ document_ingestion.py ──→ ChromaDB edu_documents (课程介绍向量)
    │
    └──→ create_smart_edu.py ──→ ChromaDB smart_edu (实体名称向量)
                              └──→ entity_mapping 表 (同义词映射)
```

**查询时的完整流程**：

```python
async def rag_process(user_query, session_id):
    # 0. 加载对话历史（通过 thread_id=session_id 从 InMemorySaver 读取）
    agent_config = {"configurable": {"thread_id": session_id}}
    
    # 1. Agent 决策（DeepSeek LLM 自主判断调用哪些工具）
    #    - 无关问题 → 直接拒绝
    #    - 需精确数据 → 调 neo4j_query
    #    - 需背景知识 → 调 document_retrieval
    #    - 两者都要 → 并行调用
    
    # 2. 实体对齐（通用前置步骤）
    #    - 先查 MySQL entity_mapping 表（同义词缓存）
    #    - 未命中 → ChromaDB smart_edu 向量相似度检索
    #    - 对齐后写入 entity_mapping 表（自动学习）
    aligned_entities = entity_alignment(user_query)
    
    # 3. 并行检索
    structured_results = await neo4j_query(aligned_entities)   # Neo4j 精确查询
    unstructured_results = await document_retrieval(user_query) # ChromaDB 语义检索
    
    # 4. 结果融合（LLM 基于 ToolMessage 中的真实数据生成）
    response = await llm.generate_with_tools(user_query, structured_results, unstructured_results)
    
    # 5. 保存状态到 InMemorySaver
    return response
```

**关键设计**：
- **不是先分类再分发**：LLM 自主决策调用哪些工具，而非硬编码 if-else
- **实体对齐是通用前置**：无论走 Neo4j 还是 ChromaDB，涉及实体名时都会先对齐
- **ToolMessage 锚定真实数据**：工具返回的数据强制进入 LLM 上下文，防止编造

## 六、常见问题与解决方案

### 6.1 回答暴露技术细节

**问题**：Agent说"让我查询一下数据库..."

**解决**：在系统提示词中明确禁止
```python
【重要限制】
- 严禁说："让我查询..."、"我需要..."、"正在检索..."
```

### 6.2 检索结果不相关

**问题**：检索到的内容与用户问题无关

**解决**：
1. 优化嵌入模型（换更好的模型）
2. 添加查询重写
3. 调整检索阈值
4. 添加重排序

### 6.3 LLM幻觉

**问题**：LLM编造不存在的信息（如虚构课程、错误价格）

**五层防护体系**：

| 层级 | 机制 | 实现方式 | 作用 |
|------|------|---------|------|
| **数据层** | RAG 检索增强 | Neo4j + ChromaDB 双路召回 | LLM 基于真实数据生成，不靠参数记忆 |
| **指令层** | 系统提示词约束 | "严禁编造：不要创造数据库中不存在的课程" | 从行为上抑制幻觉 |
| **对齐层** | 实体对齐 | 同义词表 + 向量相似度匹配 | 确保查询实体准确，避免因名称错误导致空结果 |
| **执行层** | Cypher 语法检查 | LLM 检查 + try-except 捕获 | 防止错误查询返回空，迫使 LLM 换策略而非编造 |
| **上下文层** | ToolMessage 锚定 | 工具返回数据强制注入 LLM 上下文 | LLM 生成被"锚定"在检索结果上 |

**关键原理**：
```
用户问："Redis多少钱？"
    ↓
Agent 调用 neo4j_query → 返回 ToolMessage: {"price": 299}
    ↓
LLM 看到的上下文包含 ToolMessage（真实数据）
    ↓
LLM 基于 299 生成回答 → "Redis课程299元"
    ↓
不会编造成 199 元，因为上下文里明确写了 299
```

**注意**：`ToolMessage` 对前端隐藏（用户看不到），但对 LLM **完全可见**，这是防止幻觉的核心设计。

### 6.4 响应速度慢

**问题**：用户等待时间过长

**解决**：
1. 流式输出（SSE）
2. 缓存热门查询结果
3. 异步并行检索
4. 优化模型调用（使用更快的模型或缓存）

## 七、可复用的代码模板

### 7.1 Agent初始化模板

```python
# tools/agent/__init__.py

from langchain.agents import create_agent
from langchain_deepseek import ChatDeepSeek
from langchain.tools import tool

# 导入工具函数
from .tools_def import (
    entity_alignment,
    document_retrieval, 
    neo4j_query
)
from .schema import (
    EntityAlignmentParams,
    DocumentRetrievalParams,
    Neo4jQueryParams
)
from .prompts import system_prompt_template

def get_agent(neo4j_schema_func):
    """创建Agent"""
    # 初始化LLM
    llm = ChatDeepSeek(model="deepseek-chat")
    
    # 包装工具
    tools = [
        tool(entity_alignment, args_schema=EntityAlignmentParams),
        tool(document_retrieval, args_schema=DocumentRetrievalParams),
        tool(neo4j_query, args_schema=Neo4jQueryParams),
    ]
    
    # 构建系统提示词
    system_prompt = system_prompt_template.format(
        neo4j_schema=neo4j_schema_func()
    )
    
    # 创建Agent
    agent = create_agent(
        model=llm,
        tools=tools,
        system_prompt=system_prompt
    )
    
    return agent
```

### 7.2 向量检索工具模板

```python
# tools/tools_def.py

from config.dependency import embedding_model, document_collection

def document_retrieval(query_text: str, top_k: int = 3):
    """
    从文档知识库检索相关信息
    
    Args:
        query_text: 查询文本
        top_k: 返回结果数量
    
    Returns:
        相关文档列表
    """
    # 生成查询向量
    query_embedding = embedding_model.embed_query(query_text)
    
    # 向量检索
    results = document_collection.query(
        query_embeddings=[query_embedding],
        n_results=top_k,
        include=["documents", "metadatas", "distances"]
    )
    
    # 格式化结果
    documents = []
    for i in range(len(results["ids"][0])):
        doc = {
            "content": results["documents"][0][i],
            "metadata": results["metadatas"][0][i],
            "score": 1 - results["distances"][0][i]
        }
        documents.append(doc)
    
    return documents
```

### 7.3 数据导入脚本模板

```python
# data/document_ingestion.py

import chromadb
from config.config import VECTOR_STORE_DIR, EMBEDDING_MODEL_PATH
from sentence_transformers import SentenceTransformer

def ingest_documents(documents, collection_name="documents"):
    """
    将文档导入ChromaDB
    
    Args:
        documents: 文档列表 [{"id": "", "content": "", "metadata": {}}]
        collection_name: 集合名称
    """
    # 初始化
    client = chromadb.PersistentClient(path=str(VECTOR_STORE_DIR))
    collection = client.get_or_create_collection(collection_name)
    
    # 加载嵌入模型
    model = SentenceTransformer(str(EMBEDDING_MODEL_PATH))
    
    # 批量处理
    batch_size = 32
    for i in range(0, len(documents), batch_size):
        batch = documents[i:i+batch_size]
        
        ids = [doc["id"] for doc in batch]
        contents = [doc["content"] for doc in batch]
        metadatas = [doc["metadata"] for doc in batch]
        
        # 生成嵌入
        embeddings = model.encode(contents).tolist()
        
        # 入库
        collection.add(
            ids=ids,
            documents=contents,
            metadatas=metadatas,
            embeddings=embeddings
        )
    
    print(f"成功导入 {len(documents)} 条文档")
```

## 总结

设计大模型项目的核心要点：

1. **明确边界**：解决一个具体问题，不贪大求全
2. **数据先行**：先盘点数据，再设计能力
3. **隐藏复杂性**：用户只应看到友好交互，不应看到技术过程
4. **分层设计**：数据层、检索层、生成层分离，便于迭代
5. **提示词工程**：系统提示词要包含角色、流程、示例、约束
6. **渐进迭代**：先跑通基础流程，再逐步优化

这份指南可以作为其他大模型项目的参考模板，根据具体场景调整即可。
